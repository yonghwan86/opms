```
## 그래프 날짜 범위 아이디어 검토

아이디어 자체는 매우 좋아요. 구체적으로 분석해볼게요.

**현재 구조의 문제**
```
3.27 선택 → 고정 +14일 → 4.10까지 표시 (너무 김)
3.13 선택 → 고정 +14일 → 3.27까지만 표시 (공표일 변화가 안 보임)
```

**제안하는 구조**
```
기준: 마지막 공표일 + 3일
3.13 선택 → 공표일: [3.13, 3.27] → 마지막 공표일 3.27 + 3일 → 3.30까지
3.27 선택 → 공표일: [3.27] → 마지막 공표일 3.27 + 3일 → 3.30까지
4.10 추가 후 3.13 선택 → 공표일: [3.13, 3.27, 4.10] → 4.13까지
```

**좋은 점 ✅**
- 선택한 공표일 이후의 모든 공표일 변화가 자동으로 그래프에 반영됨
- 불필요하게 긴 미래 구간이 사라짐
- 공표일이 늘어날수록 그래프가 자연스럽게 확장되는 구조

**고려해야 할 엣지케이스 ⚠️**
```
케이스 1: 가장 최신 공표일 선택 시
→ 이후 공표일 없음 → 마지막 공표일(=선택일) + 3일만 표시
→ 계단 전환이 보이지 않는 문제는 여전히 존재
→ 이 경우만 +7일 정도로 예외 처리 고려 필요

케이스 2: 공표일이 1개뿐인 경우
→ 선택일 + 3일만 표시 → 그래프가 너무 짧아질 수 있음
→ 최소 표시 기간(예: 최소 2주) 하한선 필요

케이스 3: 백엔드 endDate 연동
→ 현재 백엔드가 selectedDate + 14일로 데이터를 조회함
→ 프론트에서 범위를 줄여도 백엔드 쿼리는 여전히 +14일
→ 백엔드 endDate도 같이 바꿔야 실제 데이터가 맞게 옴
→ Out of scope(백엔드 변경 없음)와 충돌 가능성
구현 방향 제안
ts// extendEnd 계산 변경
const extendEnd = useMemo(() => {
  if (!allCeilings.length || !selectedDate) return "";
  
  const selectedKey = selectedDate.replace(/-/g, "");
  
  // 선택일 이후 공표일 중 가장 마지막
  const futureCeilings = allCeilings
    .filter(c => c.effectiveDate.replace(/-/g, "") >= selectedKey)
    .sort((a, b) => b.effectiveDate.localeCompare(a.effectiveDate));
  
  const lastCeiling = futureCeilings[0]; // 가장 마지막 공표일
  
  const base = lastCeiling 
    ? new Date(lastCeiling.effectiveDate) 
    : new Date(selectedDate);
  
  base.setDate(base.getDate() + 3); // +3일
  
  // 최소 2주 보장
  const minEnd = new Date(selectedDate);
  minEnd.setDate(minEnd.getDate() + 14);
  
  const finalEnd = base > minEnd ? base : minEnd;
  
  return `${finalEnd.getFullYear()}...`;
}, [selectedDate, allCeilings]);
